Kernel Density Estimation

The same estimator, derived three independent ways, then smoothed with a better kernel
Understanding Uncertainty · lecture notes, digitized

Estimating a density sounds like it should need new machinery. It doesn't: a moving-window sample proportion, the slope of the ECDF, and an average of measurement-error bumps all arrive at the exact same formula. This lecture derives it three times, proves it is biased but consistent, and then swaps its blocky uniform kernel for a smooth Gaussian one.

Part 1 · Kernel Density Estimation

Estimating f(x) as a moving-window proportion

How would you estimate the density \(f(x) = F'(x)\)? Intuitively, you are estimating the probability that \(X\) lands in a small neighborhood around \(x\): \(x-h \lt X \lt x+h\). The sample version of that question is "what fraction of my observations fall in that neighborhood?" — a moving-window histogram.

One property of absolute value does the bookkeeping: \(|x-x_i| \lt h\) means \(x-h \lt x_i \lt x+h\) — \(x_i\) sits inside a window centered at \(x\) with half-width \(h\). The natural estimate is then a sample proportion of points inside that window:

$$\frac{\text{number of observations in } |x-X|\lt h}{N} = \frac{\sum_{i=1}^N \mathbb{I}\{|x-x_i|\lt h\}}{N}$$

A second construction: slope of the ECDF

The derivative of the CDF is the PDF, so compute the secant of the ECDF as if preparing to differentiate it:

$$\begin{alignat*}{2}
\hat f_{X,h}(x) &=& \frac{\hat F_X(x+h) - \hat F_X(x-h)}{2h}\\
&=& \frac{\frac1n\sum_{i=1}^n \mathbb{I}\{x_i\le x+h\} - \frac1n\sum_{i=1}^n\mathbb{I}\{x_i\le x-h\}}{2h}\\
&=& \frac{\sum_{i=1}^n \mathbb{I}\{x_i\le x+h\} - \mathbb{I}\{x_i \le x-h\}}{2nh}
\end{alignat*}$$

For any one \(i\), the numerator is 1 exactly when \(x+h \ge x_i\) and \(x_i > x-h\) — i.e., \(x_i\) is between \(x-h\) and \(x+h\):

$$\hat f_{X,h}(x) = \frac{\sum_{i=1}^n \mathbb{I}\{|x-x_i|\lt h\}}{2nh}$$

Identical to the moving-window count from the last section, divided by \(2nh\) instead of \(n\) — the two derivations, starting from completely different intuitions, land on the same estimator. This is the kernel density estimator, and \(h\) is its bandwidth. The uniform kernel is \(k(z) = \mathbb{I}\{|z|\lt 1\}/2\). Let \(h \to 0\) with finite data and you get \(0/0\) everywhere except exactly at the data points — which is exactly what sns.rugplot(df[var]) shows: a comb of spikes with nothing in between.

The KDE algorithm

What bandwidth to pick? Data-driven methods exist (least-squares cross-validation), but the standard, convenient starting point is Silverman's rule-of-thumb:

$$h_u = 1.84\times \mathrm{sd}(X)\times n^{-1/5}$$

Four steps compute a KDE end to end: (1) compute the Silverman bandwidth, h_u = 1.84 * X.std() * n ** (-.2); (2) build a grid from just below the sample minimum to just above the maximum, x_grid = np.linspace(X.min()-h_u, X.max()+h_u, K); (3) compute the indicator matrix via broadcasting, I = np.abs(x_grid[:,None] - X[None,:]) < h_u; (4) sum each row and normalize, f_hat = np.sum(I, axis=1) / (2*n*h_u):

def kde(X, K=20):
    n = len(X)
    h_u = 1.84 * X.std() * n ** (-0.2)
    x_grid = np.linspace(X.min() - h_u, X.max() + h_u, K)
    I = np.abs(x_grid[:, None] - X[None, :]) < h_u
    f_hat = np.sum(I, axis=1) / (2 * n * h_u)
    return x_grid, f_hat

A third construction: measurement error

A third, independent way to arrive at the same estimator: imagine each observed \(x_i\) actually carries measurement error, uniformly distributed on \((x_i-h, x_i+h)\). Each data point then contributes its own tiny density bump,

$$\hat f_{i,h}(x) = \mathbb{I}\{|x_i-x|\lt h\}/2h$$

and averaging all \(n\) bumps together:

$$\hat f_{n,h}(x) = \frac1n\sum_{i=1}^n \hat f_{i,h}(x) = \frac1n\sum_{i=1}^n \frac{\mathbb{I}\{|x_i-x|\lt h\}}{2h}$$

the exact same formula, a third time — each data point contributes \(1/n\) of a bump, and the sum of all \(n\) bumps is the density estimate.

What is the KDE estimating?

Take the expectation of the KDE itself:

$$\mathbb{E}_X[\hat f_{X,h}(x)] = \mathbb{E}_X\!\left[\frac{\hat F_X(x+h)-\hat F_X(x-h)}{2h}\right] = \frac{F_X(x+h)-F_X(x-h)}{2h}$$

Not quite a derivative yet — but taking the limit as \(h \to 0\):

$$\lim_{h\to 0}\mathbb{E}_X[\hat f_{X,h}(x)] = F_X'(x) = f_X(x)$$

The KDE estimates the true density in the limit as the bandwidth shrinks to zero. The probability that \(X\) equals \(x\) exactly is still zero — the same lesson as the vanishing-grid puzzle from the previous lecture — but the density still captures the local behavior of \(X\) near \(x\), just not at a single point.

Bias, consistency, and the bias-variance trade-off

In practice, \(h\) is set by the Silverman plug-in, not literally 0 — it scales like \(n^{-1/5}\). That means the KDE is biased: its expectation does not equal the true density exactly. But because \(h_n \to 0\) as \(n \to \infty\), that bias shrinks to zero as the sample grows — the KDE is a consistent estimator of \(f_X(x)\), even though it is not unbiased at any finite \(n\).

Nudge the bandwidth and watch the classic bias-variance trade-off: large \(h\) averages over large neighborhoods, so the curve barely moves as the data changes (low variance, high bias); small \(h\) averages over tiny neighborhoods, so the curve swings wildly with every new sample (high variance, low bias). The Class 07 lab's regime-preset widget makes this trade-off directly draggable — applied machine learning is, in large part, judgment about exactly this trade-off.

Part 2 · The Gaussian Kernel

Constructing the exponential function

Start with the constant function \(e_0(x) = 1\) and integrate repeatedly: \(e_1(x) = \int 1\,dx = 1+x\), \(e_2(x) = \int (1+x)\,dx = 1+x+\tfrac{x^2}2\), and in general \(e_k(x) = 1+x+\tfrac{x^2}2+\cdots+\tfrac{x^k}{k!}\). Continue forever:

$$e_\infty(x) = 1+x+\frac{x^2}{2}+\cdots+\frac{x^k}{k!}+\cdots$$

This infinite series defines the exponential function, \(e^x\). Differentiating term by term reproduces the same series, so \(D_x e^x = e^x\) — it is its own derivative, hence strictly increasing. As \(x\to\infty\) every term is positive and the sum diverges, so \(e^x \to \infty\); as \(x \to -\infty\), \(e^x = 1/e^{-x} \to 1/\infty = 0\). Increasing and bounded below by 0 forces \(e^x \ge 0\) everywhere.

Example: the sigmoid/logistic distribution

One of data science's most common functions:

$$F(z) = \frac{1}{1+e^{-z}} = \frac{e^z}{e^z+1}$$

Its density has a self-referential form that makes it cheap to compute once you already have \(F\):

$$f(z) = \left(1-\frac{e^{-z}}{1+e^{-z}}\right)\frac{e^{-z}}{1+e^{-z}} = \bigl(1-F(z)\bigr)F(z)$$

The Gaussian kernel

One of the single most important functions in probability:

$$\phi(z) = \frac{1}{\sqrt{2\pi}}\exp\!\left(\frac{-z^2}{2}\right)$$

the Gaussian kernel, standard normal density, or simply "a Gaussian" — the equation for the bell curve. Since \(-z^2\) is always negative, \(\phi(z)\) is maximized at \(z=0\) (value \(1/\sqrt{2\pi}\)) and decays quickly as \(|z|\) grows; it is non-negative everywhere since \(\exp(w) \ge 0\); and it integrates to exactly 1 over the whole real line.

From uniform to gaussian kernels

The KDE derivation used the uniform kernel \(k_u(z) = \mathbb{I}\{|z|\lt 1\}/2\) inside

$$\hat f_{n,h}(x) = \frac1n\sum_{i=1}^n \frac1h\,k_u\!\left(\frac{x-x_i}{h}\right)$$

a well-behaved estimator, but a blocky one — every window edge is a discontinuity. Swap \(k_u\) for the Gaussian kernel \(\phi\) instead and the same formula smooths out completely:

$$\hat f_{n,h}(x) = \frac1n\sum_{i=1}^n \frac1h\,\phi\!\left(\frac{x-x_i}{h}\right) = \frac{1}{nh\sqrt{2\pi}}\sum_{i=1}^n \exp\!\left(-\frac{(x-x_i)^2}{2h^2}\right)$$

This is exactly what Seaborn's kdeplot computes by default. The same measurement-error story from Part 1 applies here too, with a Gaussian in place of a uniform: each \(x_i\) is measured with Gaussian error of bandwidth (standard deviation) \(h\), contributing its own bell-shaped bump,

$$\hat f_{i,h}(x) = \frac{1}{h\sqrt{2\pi}}\exp\!\left(-\frac{(x_i-x)^2}{2h^2}\right)$$

and averaging all \(n\) bumps gives back the same estimator. Swapping kernels only changes one line of the algorithm from Part 1 — the indicator-and-count becomes an exponential-and-sum:

def kde_gaussian(X, K=35):
    n = len(X)
    h_s = 1.06 * X.std() * n ** (-0.2)
    x_grid = np.linspace(X.min() - h_s, X.max() + h_s, K)
    Z = (x_grid[:, None] - X[None, :]) / h_s
    f_hat = np.sum(np.exp(-Z**2 / 2), axis=1) / (n * h_s * np.sqrt(2 * np.pi))
    return x_grid, f_hat

Same shape as the uniform-kernel function from Part 1: a grid, a broadcasted distance matrix, a per-point contribution summed across rows. Only the contribution changes — from a hard \(\mathbb{I}\{\cdot\}/2\) cutoff to the smooth Gaussian \(\phi(z) = \exp(-z^2/2)/\sqrt{2\pi}\).

Silverman's plug-in for the Gaussian

Changing the kernel changes the optimal plug-in bandwidth. For the Gaussian, the standard choice is:

$$h_s = 1.06\times \mathrm{sd}(X)\times n^{-1/5}$$

A slightly more robust variant guards against a heavy-tailed sample inflating \(\mathrm{sd}(X)\) by also considering the IQR:

$$h_s = 0.9\times \min\!\left\{\mathrm{sd}(X),\, \frac{\mathrm{IQR}}{1.34}\right\}\times n^{-1/5}$$

FAQ

Why do three completely different derivations (moving window, ECDF slope, measurement error) all land on the same formula?

Because they are the same idea described in three vocabularies. Each one asks, in its own language, "how much probability mass is within \(h\) of \(x\)?" — a window count, a secant slope, and a stack of uniform error bumps are three ways of counting the identical set of nearby points and normalizing the same way.

Why is a biased estimator still useful?

Because bias is not the only property that matters — consistency (the bias vanishing as \(n\) grows) matters just as much in practice. The KDE trades a small, known, shrinking bias for a density estimate that would otherwise be impossible to compute at all without assuming a parametric form.

Why does the Gaussian kernel need a different Silverman constant (1.06) than the uniform kernel (1.84)?

Because the optimal bandwidth balances a kernel-specific bias-variance trade-off, and different kernel shapes spread their probability mass differently around each data point. A Gaussian kernel's smooth tails need a different scaling constant than a uniform kernel's hard cutoff to hit the same optimality criterion.
