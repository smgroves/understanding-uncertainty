Wrangling & Robust Statistics

From messy real listings to summary statistics that survive the mess
Understanding Uncertainty · a build-it-from-scratch walkthrough

A used-car listing does not arrive ready to average. One row in this lab's data is a salvage-title Peterbilt truck with 1,017,936 miles on the odometer and a color field that just says "color:" — someone left the form blank and the scraper kept the label. Before any statistic can be trusted, someone has to decide what each recorded value actually means. That decision made, one lesson repeats across every variable in this dataset: the mean trusts every observation equally, and a single wrong one can wreck it. The median barely notices.

Loading a dataset is the easy part; trusting it is the work. The wrangling notebook lays out a fixed sequence — read the file, check the schema, verify types, handle missing values, filter, save — and this dataset needs every step. Its condition column mixes units freely, so "2 days left" and "21 hours left" and "34 minutes" all describe the same underlying quantity — time left in the auction — in three different disguises, and a handful of rows just say "Listing Expired". None of that is usable until it is cast to one common numeric unit:

def hours_left(raw):
    n, unit = raw.split()[:2]
    n = float(n)
    if unit.startswith('day'):
        return n * 24
    if unit.startswith('hour'):
        return n
    if unit.startswith('minute'):
        return n / 60
    return None   # "Listing Expired" and anything else unrecognized

Missing values are the harder problem, because they rarely announce themselves. A blank cell is easy to spot; a disguised missing value is not. This dataset's color column uses the literal string "no_color" for some rows and the leftover label "color:" for others — two different sentinels for the same underlying fact, that no color was recorded. The price column goes further and encodes "unknown" as the number 0, which is silently valid input to a mean() call and will never raise an error. Before any statistic runs, every variable needs the same question asked of it: what does a missing observation look like here, and have I found all of them?

Check your understanding

Show answer

Too low. Treating four missing prices as literal zeros pulls the sum down while the count of 250 stays the same, so the mean is biased toward zero. Excluding them here raises the mean from about $18,246 to about $18,543 — a small shift because n is large and only 4 of 250 rows are affected, but the direction is not a coincidence: coding "missing" as the smallest possible number always drags the mean down.

Mean and variance

A statistic is just a function that turns a list of numbers into one number. The sample mean is the most familiar: add everything up and divide by how many there are.

$$M(X) = \frac{x_1 + x_2 + \cdots + x_n}{n} = \frac{1}{n}\sum_{i=1}^n x_i$$

The sample variance measures spread — the average squared distance of each point from that mean — and its square root is the standard deviation:

$$V(X) = \frac{1}{n}\sum_{i=1}^n \bigl(x_i - M(X)\bigr)^2, \qquad \mathrm{sd}(X) = \sqrt{V(X)}$$

Drag the slider below to change the truck's mileage — everything else in the toy set of five stays fixed. Watch the mean and the standard deviation move as the one extreme value moves.

The value that doesn't flinch

  • Pair · 3 minCompare: the median stays pinned at 41 for any fifth value at or above 41 — whether it lands between the other cars or shoots past all of them, 41 keeps its middle rank. Only when the fifth value drops below 41 does it displace 41 from the middle, and the median moves with it.
  • Share · 2 minAgree on the general rule this implies — it is about rank, not magnitude — then check it against the definition below.

The median answers a different question than the mean: not "what is the total, divided evenly," but "what value sits in the middle, with half the data above and half below." Sort the data first; for an odd count the median is the middle entry, and for an even count it is the average of the two middle entries:

def median(X):
    values = sorted(X)      # rank is only meaningful once sorted
    n = len(values)
    mid = n // 2             # middle index (odd n), or right-of-center (even n)
    if mid != n / 2:          # true exactly when n is odd
        return values[mid]
    return (values[mid - 1] + values[mid]) / 2   # even n: average the two middles

Run this on the toy set sorted as \(\{12, 34, 41, 58, 1018\}\): \(n = 5\) is odd, so the median is the third entry, 41 — however far out the truck sits, it only ever occupies one end of the sorted list and never touches the middle. That is the entire mechanism behind "the median doesn't flinch": moving the largest value larger never changes which value is in the middle.

Check your understanding

Add a sixth car at 1500 (thousand miles) to the toy set, so it becomes \(\{12, 34, 41, 58, 1018, 1500\}\), n = 6. What is the median now, and why does the rule change?

Show answer

The median is \((41 + 58)/2 = 49.5\). With n = 6 there is no single middle element — index 2 and index 3 (0-indexed) both straddle the center — so the convention averages them. Both middle values, 41 and 58, are still comfortably ordinary cars; the two extreme trucks at the top of the sorted list still do not touch the median.

Quantiles and the IQR

Think · Pair · Share · generalizing the median
  • Think · 2 minThe median is "the value with half the data below it." What single number would you compute to answer "the value with a quarter of the data below it"? Sketch the recipe before reading on.
I've done the Think step — reveal Pair & Share
  • Pair · 3 minCompare recipes: sort the data, then walk 25% of the way into the sorted list instead of 50%. The median is this same operation at 50% — at least for an odd-length sample, as the next paragraph makes precise.
  • Share · 2 minAgree on what "25% of the way in" should mean for a list of 250 real prices, then test it in the widget below.

The \(f\)-th quantile generalizes the median to any split point: the value with a fraction \(f\) of the data at or below it. Sort the data, then read off the entry at the right index:

$$q_f = X_{(\lceil n f \rceil)}, \qquad \text{index} = \max\bigl(\lceil n f \rceil - 1,\ 0\bigr)$$

For an odd-length sample the median is exactly \(q_{.5}\) — both land on the same middle entry. For an even-length sample the two conventions can differ slightly: median() averages the two middle entries, while this quantile() picks a single sorted entry and never averages. On the 250-listing price sample that gap is real: the median is $16,250, but \(q_{.5}\) computed this way lands on $16,000, one order-statistic short. Two quantiles together give a robust measure of spread, the interquartile range: the gap between the 75th and 25th percentiles, which brackets the middle half of the data and — like the median — barely reacts to a single extreme listing.

$$\mathrm{IQR} = q_{.75} - q_{.25}$$

Taming the long tail

I've done the Think step — reveal Pair & Share
  • Pair · 3 minCompare answers. A transform that shrinks large gaps between big numbers more than small gaps between small numbers — like a logarithm — should pull the scattered tail in and spread out the crowded wall.
  • Share · 2 minAgree on what could go wrong with a logarithm specifically, then check the widget below.

Almost every real numeric variable has a long tail — a few values much larger than the rest — and price is no exception. The standard fix is the natural logarithm, np.log(df[var]), which compresses large values far more than small ones and turns a lopsided histogram nearly symmetric. Its flaw is that it is undefined at zero and for negative numbers — exactly the disguised-missing zeros this dataset already has.

The fix for the fix is the inverse hyperbolic sine, or arcsinh, which behaves like a logarithm for large values but is defined everywhere, including zero and negative numbers, where it evaluates to 0 and to negative outputs respectively:

$$i(x) = \ln\!\left(x + \sqrt{x^2 + 1}\right)$$

Toggle between raw, log, and arcsinh price below. The four price = 0 listings vanish from the log histogram (their transformed value is undefined) but stay put, at zero, under arcsinh.

Every statistic so far has summarized one variable at a time. The natural next question is conditional: how does price behave within a category? FAQ

Why does the median only ever move to another observed value?

Because it is defined by rank, not by arithmetic — it is either literally one of the data points (odd \(n\)) or the average of two adjacent ones (even \(n\)). Nothing outside the two middle ranks can influence it, no matter how extreme.

Is the median always the better choice?

No. The mean uses every value's exact magnitude, which is exactly what you want when magnitudes carry real information you do not want to throw away — total revenue is a sum of means, not medians. The median is the better choice specifically when a few extreme or erroneous values would otherwise dominate a statistic meant to describe the typical case.

Why not just delete every outlier before computing anything?

Because "outlier" is a judgment call, not a fact visible from one number. The salvage truck's mileage is extreme but plausibly real; a price of exactly $0 is extreme and almost certainly wrong. Wrangling is the process of making that judgment call explicit, variable by variable, instead of encoding it silently into whichever statistic you happened to compute.

Does arcsinh change which car is more expensive than which?

No — it is a strictly increasing function, so it preserves every ordering in the data. It changes the visual and numerical scale, compressing large gaps between expensive cars, without changing any comparison between two listings.
