Coins, Trials, and the Binomial

How one 0/1 flip becomes a whole distribution of counts
Understanding Uncertainty · a build-it-from-scratch walkthrough

Flip a coin once and you get a single 0 or 1 — a Bernoulli trial. Flip it \(n\) times and count the heads, and that count is no longer a single number but a distribution over the values \(0, 1, \dots, n\). That distribution is the binomial, and it is the workhorse behind every "how many out of how many" question — how many patients survive, how many packets drop, how many emails are spam.

Built for Class 03 of Understanding Uncertainty. Data: the Heart Failure Clinical Records dataset (uu_fa26/class_02/heart_failure_clinical_records_dataset.csv), using the binary DEATH_EVENT column — one 0/1 outcome per patient, \(n = 299\).

From one flip to a count

A single trial is the simplest random variable there is. It takes the value 1 with probability \(p\) and 0 with probability \(1-p\), and that is its entire distribution:

$$P(X_i = 1) = p, \qquad P(X_i = 0) = 1 - p, \qquad \mathbb{E}[X_i] = p$$

Because the outcome is already a 0 or a 1, each trial doubles as an indicator: \(X_i = \mathbb{I}\{\text{trial } i \text{ succeeds}\}\). In the heart-failure data, \(X_i = 1\) means patient \(i\) died during follow-up. The sample proportion of deaths is just the average of these indicators, and averaging 0/1 values is the same as counting the 1s and dividing:

$$\hat{p} = \frac{1}{n}\sum_{i=1}^{n} x_i = \frac{\#\{\,x_i = 1\,\}}{n} = \frac{96}{299} \approx 0.321$$

Now define the count \(X = \sum_{i=1}^{n} X_i\), the sum of \(n\) trials. If the trials are iid — same \(p\) every time, no trial nudging another — then \(X\) follows a Binomial\((n, p)\) distribution. The whole lab is the study of that one sum: what values it takes, how likely each is, and what happens as \(n\) grows.

Note · Bernoulli is Binomial with n = 1

A single Bernoulli trial is exactly Binomial\((1, p)\): the count of successes in one trial is 0 or 1. So everything below reduces to the coin-flip case when you slide \(n\) down to 1 in the explorer above.

The binomial PMF

Any one sequence with \(k\) successes and \(n-k\) failures has probability \(p^k(1-p)^{n-k}\) — multiply the per-trial probabilities, since the trials are independent. The catch is that many different sequences give the same total \(k\). The binomial coefficient \(\binom{n}{k}\) counts exactly how many, and multiplying gives the probability of the count:

$$P(X = k) = \binom{n}{k}\, p^{k}\,(1-p)^{\,n-k}, \qquad \binom{n}{k} = \frac{n!}{k!\,(n-k)!}$$

Read it left to right: choose which \(k\) of the \(n\) trials succeed \(\big(\binom{n}{k}\big)\), pay \(p\) for each success \((p^k)\), and pay \(1-p\) for each failure \(\big((1-p)^{n-k}\big)\). The mean and variance fall straight out of the sum-of-trials view, because expectation adds and, for independent trials, so does variance:

$$\mathbb{E}[X] = n p, \qquad \operatorname{Var}(X) = n p (1-p)$$

In code the PMF is a few lines with no libraries. The one subtlety is the coefficient: build it with a multiplicative loop instead of computing giant factorials that overflow.

def n_choose_k(n, k):
    if k < 0 or k > n:
        return 0.0
    k = min(k, n - k)                 # C(n,k) == C(n,n-k)
    c = 1.0
    for i in range(k):
        c = c * (n - i) / (i + 1)     # build the coefficient step by step
    return c

def binom_pmf(n, p, k):
    if k < 0 or k > n:
        return 0.0
    return n_choose_k(n, k) * p ** k * (1 - p) ** (n - k)

At \(p = 0.5\) the bar at \(k = 5\) is tallest, and the plot is mirror-symmetric because a success and a failure are equally likely, so \(P(X=k)=P(X=n-k)\). Lowering \(p\) to 0.3 makes failures more likely, so mass slides left toward smaller counts; the peak moves to around \(k = n p = 3\) and the shape becomes right-skewed.

The PMF describes the count after a fixed number of trials. But you can also watch the process unfold trial by trial. A sample path plots the running proportion of successes against the trial number: after \(t\) trials it is the count so far divided by \(t\). Each new flip nudges it up or down.

Play the widget and watch several paths at once. Early on, one lucky flip swings the proportion wildly — with only 3 trials, one extra head moves it by a third. As \(t\) grows, each flip matters less, and every path is squeezed toward the dashed line at \(p\). That squeeze is the law of large numbers, and it is the reason a larger sample gives a more trustworthy estimate.

Running proportion · one path per color
Each colored line is one sequence of iid \(p\)-coins; its height at trial \(t\) is the proportion of successes so far. The dashed line is the true \(p\). Press Play to reveal trials, or Step through them; Reseed draws fresh sequences.
The spread between paths shrinks like \(1/\sqrt{t}\): to halve the wobble you need four times the trials. This is the same reason the binomial's proportion gets sharper as \(n\) grows even though its count spreads wider.

Assignment

Implement the binomial PMF from scratch — no numpy, and no math.comb shortcut. You are given the data loader, the input/output plumbing the autograder speaks, and the check that \(p\) is a valid probability. You write two functions.

What you build

  • n_choose_k(n, k) — the binomial coefficient \(\binom{n}{k}\) via a multiplicative loop; return 0 when \(k<0\) or \(k>n\).
  • binom_pmf(n, p, k) — return \(P(X=k)=\binom{n}{k}p^k(1-p)^{n-k}\), handling the edges \(p=0\) and \(p=1\).

I/O protocol

The grader launches python3 binomial.py and sends one query per line on stdin: <n> <p> <k>. For each line you print one line back:

OK: <P(X=k)>         # the probability, 6 decimals
ERR: invalid p        # if p is outside [0, 1]

The \(p\)-range check is already written in main(); do not print extra lines or debug text to stdout (use stderr for debugging).

FAQ

Why not just use math.comb or scipy.stats.binom?

You may in real work. The point of the assignment is to see that the coefficient is a count of orderings and the PMF is one line of arithmetic — so that when a library gives a surprising number you can check it by hand.

Why build n_choose_k with a loop instead of factorials?

\(n!\) overflows fast — \(171!\) is already infinity in a float. The multiplicative form keeps every partial product near 1, so it stays accurate for far larger \(n\). It is also how the samplers and the page widgets compute it, so all three agree to the last decimal.

The binomial's spread grows with \(n\), but the sample paths get calmer. Which is it?

Both, for different quantities. The count \(X\) has standard deviation \(\sqrt{np(1-p)}\), which grows with \(n\). The proportion \(X/n\) has standard deviation \(\sqrt{p(1-p)/n}\), which shrinks. The count spreads out; the proportion sharpens. The law of large numbers is a statement about the proportion.

When does the iid assumption break?

When trials influence each other (drawing cards without replacement) or when \(p\) drifts between trials (a machine that wears down). Then the count is not binomial, and the formula gives wrong probabilities even though it still returns a number.

Why does the PMF sometimes sum to 0.999999 instead of 1?

Rounding. Each probability is printed to six decimals, and summing many rounded values loses a little in the last place. The exact sum is 1; the displayed sum is 1 up to floating-point and rounding error.
